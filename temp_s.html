Implementation
Time to Market — Two Approaches
Options for enabling the T+2 shifted comparison model for Almea legal entities in Encore
Option A
Backend Exception
T+2 comparison logic hardcoded as an exception rule in the balancing engine
⚙️
How it works: Almea LEs are hardcoded in backend config. Balancing engine detects these LEs and applies the shifted window — GMI EOD (T) vs PTP EOD (T+1).
🔧
Change required: Backend code change to balancing engine. Each new LE requires a code deployment.
📋
Trade breaks: Auto-suppress one-day-old unmatched GMI trades for flagged LEs.
PROS
Faster initial delivery — no UI changes
Centralised control — less misconfiguration risk
Suitable for LVT and initial client activity
CONS
Each new LE requires code change + deployment
Not self-service — ops depends on tech
Not scalable long-term
VS
Option B
User Screen Tagging
Self-service UI for ops teams to tag legal entities into the T+2 balancing model
🖥️
How it works: Configuration screen in Encore allows users to tag specific LEs as "T+2 model". Balancing engine reads config dynamically.
🏷️
Change required: New UI config screen, backend dynamic config, validation + audit trail for LE assignments.
📋
Trade breaks: Same auto-suppression, driven by UI tags instead of hardcoded rules.
PROS
Self-service — ops onboards new LEs without tech
Scalable for BAU growth
Full audit trail
CONS
Longer build time — UI dev + testing
Misconfiguration risk if uncontrolled
May not be ready for LVT timeline
💡
Recommended path: Start with Option A (Backend Exception) for LVT and initial client activity — faster time to market. Transition to Option B (User Screen Tagging) as the strategic solution once Almea LE count grows and BAU requires self-service.
