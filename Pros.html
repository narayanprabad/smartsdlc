Implementation
Time to Market — Two Approaches
Options for enabling the T+2 shifted comparison model for XYZ legal entities in Encore
Option A
Backend Exception
T+2 comparison logic hardcoded as an exception rule in the reconciliation engine
⚙️
How it works: XYZ legal entities are hardcoded in backend config. The rec engine detects these LEs and automatically applies the shifted window — YYY EOD (T) vs PTP EOD (T+1).
🔧
Change required: Backend code change to the reconciliation engine. Each new LE requires a code deployment.
📋
Trade breaks: Auto-suppress one-day-old unmatched YYY trades for flagged LEs, separating expected timing lags from genuine exceptions.
PROS
Faster initial delivery — no UI changes needed
Centralised control — less risk of misconfiguration
Suitable for LVT and initial client activity
CONS
Every new LE requires a code change and deployment
Not self-service — ops teams depend on tech for changes
Not scalable long-term as LE count grows
VS
Option B
User Screen Tagging
Self-service UI for operations teams to tag legal entities into the T+2 reconciliation model
🖥️
How it works: A configuration screen in Encore allows users to tag specific LEs as "T+2 model". The rec engine reads this config dynamically and applies the shifted comparison window.
🏷️
Change required: New UI configuration screen for LE tagging, backend changes to read dynamic config, and validation / audit trail for LE assignments.
📋
Trade breaks: Same auto-suppression logic, but driven by LE tags set through the UI rather than hardcoded rules.
PROS
Self-service — ops teams onboard new LEs without tech
Scalable — supports BAU growth as XYZ expands
Audit trail — visibility of who tagged which LE and when
CONS
Longer build time — requires UI development and testing
Risk of misconfiguration if tagging is not controlled
May not be ready for LVT timeline
💡
Recommended path: Start with Option A (Backend Exception) for LVT and initial client activity — faster time to market with controlled risk. Transition to Option B (User Screen Tagging) as the strategic long-term solution once XYZ LE count grows and BAU operations require self-service capability.
